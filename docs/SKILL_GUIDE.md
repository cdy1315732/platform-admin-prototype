# Platform Admin Prototype Skill 使用指南

## 1. Skill 定位

`platform-admin-prototype` 用于把 PC 后台、B 端业务系统和 Admin 管理台需求转成可交互前端原型。

它不是一个通用网页生成 Skill，也不是只输出线框图或静态 HTML。它的目标是依据仓库内已经确认的组件、组合模式、页面结构和视觉规范，生成可以直接运行和继续开发的 Vue 3 后台页面。

## 2. 触发条件

以下需求可以触发 Skill：

- 用户提供 PRD，并要求生成后台页面或原型。
- 用户描述列表、详情、表单、弹窗、抽屉或工作台需求。
- 用户提供后台页面截图，希望按本地组件规范还原。
- 用户要求把某个业务流程做成可点击的 PC 管理台。
- 用户要求基于现有组件平台组装新页面。

显式调用示例：

```text
使用 $platform-admin-prototype 处理以下 PRD。
```

允许隐式触发的典型表达：

```text
根据这份需求做一个可交互的 PC 后台原型。
```

以下情况不应触发：

- 移动端应用
- 小程序前台
- 官网和营销页
- C 端消费产品
- 纯文档整理
- 不需要界面和交互实现的普通需求分析

## 3. 可接受的输入

Skill 可以处理以下一种或多种输入：

### PRD 或产品文档

从文档中提取：

- 页面清单
- 页面层级
- 字段和校验规则
- 表格列
- 筛选条件
- 操作按钮
- 状态枚举
- 权限规则
- 流程节点
- 弹窗和抽屉
- 成功、失败和异常反馈

### 需求描述

需求不需要先整理成完整 PRD。只要包含业务目标、使用对象和主要功能，Skill 可以补充保守的示例数据与页面结构。

### 截图或参考图

截图用于理解页面结构和业务信息层级。最终实现仍优先遵守本仓库的组件规范，不会直接复制参考图中的不一致样式。

### 结构化字段

也可以直接提供：

- 导航菜单
- 筛选项
- 表格列
- 表单字段
- 状态和颜色
- 页面按钮
- 操作权限

## 4. 执行流程

### 第一步：判断适用范围

确认任务属于 PC 后台、B 端系统或管理台。如果页面类型不明确，只有在影响产品范围时才询问用户。

### 第二步：建立需求清单

整理：

- 一级页面
- 二级页面
- 弹窗
- 抽屉
- 表格和筛选
- 表单和校验
- 业务状态
- 操作流程
- 结果反馈

### 第三步：规划信息架构

默认规则：

- 一级业务页面进入左侧导航。
- 创建、编辑和详情等按钮触发页面属于二级页面。
- 二级页面不进入左侧导航。
- 二级页面保持父级菜单选中。
- 轻量编辑使用弹窗。
- 辅助详情和长表单可以使用右侧抽屉。
- 顶部面包屑只显示必要层级。

### 第四步：选择组件与页面模板

需求映射示例：

| 需求 | 默认实现 |
| --- | --- |
| 多条件筛选 | `PlatformFilterPanel` |
| 数据列表 | `PlatformTableCard` + `PlatformDataTable` |
| 表单页面 | `PlatformFormCard` |
| 详情摘要 | `PlatformDetailSummary` |
| 新增或编辑弹窗 | `PlatformFormModalTemplate` |
| 详情抽屉 | `PlatformDetailDrawerTemplate` |
| 流程状态 | `PlatformSteps` |
| 操作记录 | `PlatformTimeline` |
| 结果反馈 | `PlatformResult` 或全局消息 |
| 图表 | 先读取 `chart-rules.md`，再进行 ECharts 选型 |

### 第五步：替换业务内容

保留组件结构和交互规则，替换：

- 页面名称
- 字段
- 表格列
- 筛选项
- 按钮
- 状态
- 模拟数据
- 业务文案

### 第六步：实现真实交互

默认需要实现：

- 导航切换
- 面包屑返回
- 输入框回显
- 搜索和重置
- 下拉默认值
- 表格勾选
- 批量操作条件显示
- 时间升降序
- 分页
- 弹窗打开和关闭
- 抽屉打开和关闭
- 表单取消和提交
- 上传可点击和禁用状态
- 开关启用和禁用状态
- 成功和错误反馈

### 第七步：浏览器验证

除非用户明确要求不验证，否则检查：

- 页面是否能启动
- 导航和二级页面是否正确
- 表单是否可以输入
- 表格和筛选是否可交互
- 弹窗和抽屉是否正常
- 页面是否存在横向溢出
- 文案是否被截断
- 组件是否重叠或压缩
- PC 宽屏布局是否合理

## 5. 默认输出

完成后通常交付：

- Vue 3 页面代码
- 本地运行地址
- 页面和交互说明
- 已使用的组件与页面模板
- 关键实现假设
- 测试、类型检查和构建结果
- 必要时提供 PC 宽屏截图

## 6. 详细功能清单

### 工作台

输入需要包含业务指标、重点事项、快捷入口、分布或趋势需求。输出可以包含：

- 指标卡片
- 待处理或异常事项列表
- 快捷操作入口
- 分类或状态分布
- 最近操作
- 条件触发的 ECharts 图表

工作台模块之间使用统一间距，不通过固定空白占位强行对齐不同高度的卡片。

### 筛选模块

输入需要描述筛选字段、字段类型、默认值和重置规则。输出需要保证：

- 筛选模块独立于表格卡片
- 字段左对齐
- 下拉项具有明确默认值
- 日期区间具有最小宽度
- 搜索和重置与字段同一基线
- 重置后恢复默认筛选和第一页

### 表格模块

输入需要描述表格列、主键、排序列、分页、批量操作和行操作。输出需要保证：

- 表格标题与表格级操作处于同一行
- 头像保持正圆
- 状态使用适合表格密度的紧凑样式
- 排序表头默认中性，点击后切换升序和降序
- 批量操作只有在勾选后显示
- 分页位于表格卡片底部
- 无数据、加载和错误状态有明确反馈

### 表单模块

输入需要描述字段、字段类型、必填、默认值、校验和提交结果。输出需要保证：

- 输入值实时回显
- 必填和校验错误可见
- 禁用字段不可编辑
- 表单操作位于底部
- 取消不会产生提交副作用
- 提交成功或失败有明确反馈
- 上传默认文件列表为空

### 详情模块

输入需要描述标题、辅助信息、摘要字段、流程、详情区块和页面操作。输出需要保证：

- 标题和操作区左对齐
- 状态使用颜色圆点加文案
- 优先级使用紧凑标签
- 流程使用步骤组件
- 操作记录使用时间线
- 详情页保持父级导航选中
- 标题操作区发生换行时增加 `12px` 间距

### 弹窗与抽屉

输入需要说明操作是否阻断主任务、表单长度和信息复杂度。默认决策：

- 短表单、确认和轻量操作使用弹窗
- 辅助详情和长表单使用右侧抽屉
- 完整创建、编辑和详情流程使用二级页面

输出需要保证弹窗居中、标题左对齐、左右内边距 `20px`，并实现关闭、取消、确认、加载和错误状态。

### 设置、分类和人员管理

输入需要描述配置分组、层级数据、角色、人员和启用禁用规则。输出可以包含：

- 页内设置菜单或标签页
- 分类树和组织树
- 添加子节点、编辑和删除
- 人员列表和角色筛选
- 启用禁用开关
- 恢复默认和保存

开关必须使用本地样式修正，不能出现圆点上下不居中或容器尺寸崩坏。

## 7. 组件层级

### 基础组件

负责单一交互或展示能力，例如输入框、按钮、标签、日期、复选框和开关。

### 表格单元格

负责表格中的稳定信息密度和对齐，例如状态、头像文本、标签、排序和行操作。

### 组合组件

封装完整业务模块，例如筛选面板、表格卡片、详情摘要和表单卡片。生成原型时优先修改 props、slots 和子组件内容，不重新设计外部结构。

### 页面模板

定义列表页、详情页、表单弹窗、详情抽屉和结果页的页面骨架。

## 8. 视觉和交互原则

### 页面布局

- 采用真实后台壳层。
- 保持稳定的侧边栏、顶部栏和内容区。
- 卡片间距通常为 `16px`。
- 页面模块不嵌套多层卡片。
- 页面标题和辅助说明不重复出现。

### 筛选

- 独立模块展示。
- 所有字段左对齐。
- 搜索和重置与字段同一基线。
- 时间区间有最小宽度。
- 下拉筛选默认显示“全部...”。

### 表格

- 表格标题和主要操作处于同一行。
- 创建、导出和添加按钮放在表格标题右侧。
- 批量操作只在勾选后显示。
- 时间排序默认不选中。
- 头像保持正圆。
- 状态和优先级使用紧凑展示方式。

### 详情

- 状态使用颜色圆点加文案。
- 优先级使用紧凑边框标签。
- 摘要字段高度克制。
- 流程使用步骤节点。
- 页面头部不使用多余返回箭头。
- 操作按钮换行时与标题保持 `12px` 间距。

### 弹窗和抽屉

- 弹窗始终居中。
- 弹窗标题左对齐。
- 弹窗左右内边距为 `20px`。
- 抽屉用于辅助详情和长表单，不替代所有二级页面。

### 上传和开关

- 上传默认没有文件。
- 上传使用灰色按钮。
- 提供 hover、active、disabled 和文件列表状态。
- 小号开关圆点上下居中。
- 禁用开关为灰色且不可操作。

## 9. 图表处理

只有需求明确包含以下内容时才进入图表流程：

- 趋势
- 对比
- 分布
- 占比
- 排名
- 漏斗
- 雷达
- 时间序列
- 统计可视化

图表流程：

1. 读取 `references/chart-rules.md`。
2. 判断数据关系和业务问题。
3. 到 ECharts 官方示例中寻找合适图形。
4. 使用 ECharts 作为图表实现。
5. 图表外部容器、标题、间距和颜色仍遵守本地规范。

没有图表需求时，不访问图表站点，也不强行添加图表。

## 10. 需求歧义

以下歧义可以自行决定：

- 精确颜色
- 阴影
- 小范围间距
- 示例数据
- 普通状态文案

以下歧义需要询问：

- 页面是否属于一级导航
- 功能是否需要权限控制
- 删除是否可恢复
- 提交是否产生真实外部副作用
- 关键业务流程是否存在审批或回退

## 11. 示例

### 列表与详情

```text
使用 $platform-admin-prototype 生成一个供应商管理后台。
一级页面包含供应商列表和分类管理。
供应商详情通过表格行操作进入二级页面。
列表支持名称、状态、分类和创建时间筛选。
```

### 表单和弹窗

```text
生成一个人员管理后台。
需要人员列表、角色筛选、添加人员表单、编辑角色弹窗、
启用禁用开关和批量停用操作。
```

### 仪表盘和图表

```text
生成运营工作台。
需要核心指标卡片、近 30 天趋势、渠道占比、异常任务列表和详情抽屉。
```

## 12. 仓库内容与 Skill 包的关系

公开仓库包含三类内容：

1. `packages/`：可直接复用和继续开发的组件、变量、注册表和页面模板源码。
2. `docs/`：供开发者和使用者阅读的详细功能、页面模式、交互和视觉规范。
3. `skills/platform-admin-prototype/`：Codex Skill 的可读源码。

`dist/skills/platform-admin-prototype.skill` 是可安装包，内部包含 Skill 指令、规范文档和必要的组件源码快照。仓库文档用于公开介绍和维护，Skill 包用于安装和执行；两者内容职责不同，但规则必须保持一致。

维护组件规范时，应同步检查：

- 组件源码是否反映新规则
- `docs/platform-admin-spec/` 是否更新
- Skill 的 `references/` 是否更新
- `.skill` 安装包是否重新打包
- README 和截图是否仍与当前实现一致
